# Internationalization (i18n) Standards

## Rule Priorities

| Priority | Category                                   | Impact   |
| -------- | ------------------------------------------ | -------- |
| 1        | Never Hardcode User-Facing Strings         | CRITICAL |
| 2        | Pluralization & ICU Message Format         | HIGH     |
| 3        | Date/Number/Currency Formatting per Locale | HIGH     |
| 4        | Translation Key Conventions                | MEDIUM   |
| 5        | RTL Support                                | MEDIUM   |
| 6        | CI Missing Key Detection                   | LOW      |
| 7        | Fallback Strategy                          | LOW      |

## Key Principles

- Never hardcode user-facing strings
- Use translation keys consistently
- Handle pluralization properly
- Format dates, numbers, and currencies per locale

## Translation Keys

```typescript
// тЬЕ Good - namespaced, descriptive keys
t("auth.login.title");
t("auth.login.submitButton");
t("auth.errors.invalidCredentials");

// тЭМ Bad - flat, unclear keys
t("login");
t("button1");
t("error");
```

## Interpolation & Pluralization

```typescript
// Interpolation
t("welcome", { name: user.name });
// "Welcome, {{name}}!" тЖТ "Welcome, John!"

// Pluralization
t("items", { count: items.length });
// count=1: "1 item"
// count=5: "5 items"
```

## Date & Number Formatting

```typescript
const formatDate = (date: Date, locale: string) => {
  return new Intl.DateTimeFormat(locale, {
    year: "numeric",
    month: "long",
    day: "numeric",
  }).format(date);
};

const formatCurrency = (amount: number, locale: string, currency: string) => {
  return new Intl.NumberFormat(locale, {
    style: "currency",
    currency,
  }).format(amount);
};

// Examples
formatDate(new Date(), "de-DE"); // "2. April 2026"
formatDate(new Date(), "en-US"); // "April 2, 2026"
formatCurrency(1234.56, "de-DE", "EUR"); // "1.234,56 тВм"
```

## Component Usage

```tsx
import { useTranslation } from "react-i18next";

export function LoginForm() {
  const { t } = useTranslation("auth");

  return (
    <form>
      <h1>{t("login.title")}</h1>
      <label>{t("login.email")}</label>
      <button type="submit">{t("login.submit")}</button>
    </form>
  );
}
```

## Do's and Don'ts

```tsx
// тЬЕ Good
<p>{t('greeting', { name: user.name })}</p>

// тЭМ Bad - hardcoded string
<p>Hello, {user.name}!</p>

// тЭМ Bad - concatenation
<p>{t('hello') + ' ' + user.name}</p>
```

## RTL (Right-to-Left) Support

```tsx
// тЬЕ Good - dynamic text direction
const isRTL = ['ar', 'he', 'fa', 'ur'].includes(locale);

<html lang={locale} dir={isRTL ? 'rtl' : 'ltr'}>

// тЬЕ Good - use logical CSS properties (RTL-safe)
// margin-inline-start instead of margin-left
// padding-inline-end instead of padding-right
<div className="ms-4 pe-2">...</div> // Tailwind: ms-* / me-* / ps-* / pe-*
```

## Gotchas

- **String concatenation breaks in RTL and gendered languages** тАФ `t('hello') + ' ' + username` fails for Arabic (RTL word order) and languages with grammatical gender. Always use interpolation: `t('hello', { name: username })`.
- **Hardcoded strings in error messages and toast notifications** тАФ these are easy to miss during i18n audits. Run a grep/lint pass for bare string literals in UI-facing code before shipping.
- **Pluralization in English тЙа pluralization elsewhere** тАФ English has 2 plural forms; Arabic has 6. Always use the i18n library's plural API, never `count === 1 ? 'item' : 'items'`.
- **Date/currency formatting in tests is locale-dependent** тАФ `new Date().toLocaleDateString()` produces different output per OS locale. Mock `Intl` or use fixed locale strings in tests.
- **Missing translation fallback throws in production** тАФ configure `missingKeyHandler` to log silently in production, not throw. Only throw in development/test.

## Next.js App Router (next-intl)

For Next.js App Router projects, use `next-intl` тАФ it integrates with React Server Components:

```typescript
// app/[locale]/layout.tsx
import { NextIntlClientProvider } from 'next-intl';
import { getMessages } from 'next-intl/server';

export default async function LocaleLayout({ children, params: { locale } }) {
  const messages = await getMessages();
  return (
    <NextIntlClientProvider messages={messages}>
      {children}
    </NextIntlClientProvider>
  );
}
```

```typescript
// Server Component тАФ no 'use client' needed
import { useTranslations } from 'next-intl';

export default function HomePage() {
  const t = useTranslations('home');
  return <h1>{t('title')}</h1>;
}
```

## ICU Message Format

For complex messages with pluralization, gender, and selects тАФ use ICU syntax:

```json
// locales/en.json
{
  "inbox": "{count, plural, =0 {No messages} one {# message} other {# messages}}",
  "greeting": "{gender, select, male {He} female {She} other {They}} joined.",
  "discount": "Save {amount, number, ::currency/USD} today!"
}
```

```typescript
t("inbox", { count: 3 }); // "3 messages"
t("inbox", { count: 1 }); // "1 message"
t("greeting", { gender: "female" }); // "She joined."
```

## CI: Missing Key Detection

Detect untranslated keys before they reach production:

```bash
# Using i18next-scanner (package.json script)
npx i18next-scanner --config i18next-scanner.config.js

# Using a simple shell check тАФ fail CI if keys differ between locales
node -e "
  const en = Object.keys(require('./locales/en/common.json'));
  const de = Object.keys(require('./locales/de/common.json'));
  const missing = en.filter(k => !de.includes(k));
  if (missing.length) { console.error('Missing DE keys:', missing); process.exit(1); }
"
```

Add to CI pipeline:

```yaml
# .github/workflows/ci.yml
- name: Check missing translation keys
  run: node scripts/check-i18n.js
```

## Fallback Strategy

- Always define a fallback locale (typically `en`)
- Missing translations should fall back silently, never throw
- Log missing translation keys in development only
